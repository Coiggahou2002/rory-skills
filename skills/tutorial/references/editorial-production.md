# Editorial Production System

This guide controls the visual quality of DOCX, HTML, PDF, and chapter visuals.

## Design System

### HTML Output

HTML uses the tutorial skill's own CSS (`templates/tutorial-style.css`):

- Clean centered report shell: sticky text navigation on the left, reading column on the right
- `clamp()` for responsive type and spacing
- Strong typographic hierarchy with serif headings and sans body
- One accent color: ink blue `#1B365D`, under 5 percent of the page
- Warm off-white background `#f5f4ed`
- Serif for titles/headings (`Noto Serif SC`, `Georgia`), sans for body (`Noto Sans SC`, system)

### PDF Output

PDF uses the Kami skill's design system:

- Warm parchment `#f5f4ed` page background
- Single ink-blue accent `#1B365D` (max 5% of document area)
- Chinese headings: TsangerJinKai02 (W05 for headings, W04 for body)
- English headings: Newsreader (serif)
- English body: Inter (sans) or Newsreader
- No italic fonts
- Warm-toned grays only (no cool/blue grays)
- Solid hex backgrounds (no `rgba()` — WeasyPrint rendering bug)
- `@page` A4 with 20mm margins
- No headers, footers, page counters, or file paths

### DOCX Output

DOCX follows the same warm palette:

- Warm parchment and ivory table shading
- Ink-blue heading accent
- No running headers or footers
- Light warm borders on tables

## Default Visual Direction

Default to a teaching report, not a decorative artifact:

- Every visible element must serve real tutorial content or navigation.
- Use strong typographic hierarchy instead of decorative frames.
- Use hairline rules and light borders; avoid hard shadows.
- Keep corner radius `6-8px` for framed visuals and tables.

Avoid:

- Purple-blue gradient defaults
- Glassmorphism, thick shadows, decorative blobs
- Emoji icons, repeated generic cards
- Background grids or paper textures in formal output
- Visible helper labels such as "generated by" or instructions inside the final artifact
- Absolute local filesystem paths in the visible source appendix

## Element Budget

Allowed persistent elements:

- Document title
- Sticky anchor-text table of contents (HTML only)
- Section headings
- Body text
- Tables, callouts, source IDs
- Chapter visuals and captions
- Source appendix with readable labels

Not allowed in final output:

- Duplicate title blocks
- Decorative subtitles not supplied by the tutorial
- Visible usage instructions
- Background patterns that do not carry information
- Extra wrappers around visuals

## Long Tutorial Length

- Chinese: `5000-10000` characters excluding source appendix.
- English: `3500-7000` words excluding source appendix.
- Default chapter count: `6-10`.
- Each chapter: `600-1200` Chinese characters or `450-900` English words.
- Opening hook: `300-600` Chinese characters or `200-400` English words.
- Final practice section: at least one applied exercise and one self-check rubric.

## HTML Report Contract

- Include a table of contents generated from headings.
- Style `nav#TOC` as a plain anchor-text menu.
- Sticky TOC on the left, reading column on the right, centered together.
- Reading column between `700px` and `860px`.
- Responsive type with `clamp()`.
- Textbook-style numbering: `第1章 章节标题` for H2, `1.1 小节标题` for H3.
- `scroll-margin-top` on headings for anchor jumps.
- For print/PDF, avoid sticky UI chrome.
- No duplicate title blocks.

## Visual Artifact Contract

Every chapter visual (Mermaid diagram) should be a teaching object:

- One visual explains one idea.
- Use varied diagram types across chapters.
- Captions tell the reader what to notice.
- Use high-resolution screenshots: `2×` scale.
- No stock-like atmosphere; the visual must reveal a real relationship, process, or model.

## Rhythm Rules

- Open with a strong hook, then slow down into explanation.
- Alternate dense teaching chapters with lighter examples.
- Avoid three consecutive sections with the same structure.
- Use callouts sparingly: checkpoint, common pitfall, what to notice.

## Production Quality Gate

Before delivery, check:

- Body length matches the requested or default length range.
- Every chapter has a visual and caption.
- Screenshots are crisp enough for PDF/Word.
- HTML tables do not overflow or squeeze text.
- Word and PDF have no visible headers or footers.
- HTML has a sticky anchor menu.
- PDF opens and images are visible.
- DOCX opens and contains embedded images.
- No absolute filesystem paths appear in final outputs.
- Typography, color, captions, and tables follow the design system.
- No visible element is decorative-only.
